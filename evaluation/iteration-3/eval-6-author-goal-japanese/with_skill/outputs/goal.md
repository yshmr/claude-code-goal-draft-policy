リポジトリ（`acme-auth-service`、`node-app`）を確認しました。認証関連のテストは `tests/auth/login.test.ts` にあり、`package.json` に `"test:auth": "jest tests/auth"` という専用スクリプトが既に定義されています。これを検証コマンドとして使います。

以下をそのまま貼り付けてください。

```
/goal tests/auth 配下の全テストが通っている状態にする。証明方法：直近のターンで
`npm run test:auth` を実行し、その出力に Jest のサマリー行として
`Tests:       2 passed, 2 total` が含まれることを示す（フィルタなしの
`tests/auth` 全体を対象とし、部分実行では満たさない）。ただし 15 ターンに達した場合、
または同じ失敗が2ターン連続で再発した場合はそこで停止し、何がブロッカーかを要約する
——このいずれかが真になった時点で完了とする。制約：`tests/` 配下のテストファイルは
一切変更・削除しないこと。各ターン `git diff --stat` を表示し、変更が `src/auth/` 配下
のみであることを示すこと。
```

選定理由：
- **検証コマンド**：`package.json` の `scripts.test:auth`（`jest tests/auth`）が既に存在するのでこれを採用しました。全体テスト（`npm test`）ではなく認証範囲に絞れるうえ、CIが無いこのフィクスチャではこれが「グリーンの定義」として最も明確です。
- **エビデンス**：`tests/auth/login.test.ts` は2件のテストなので、Jestのサマリー行`Tests: 2 passed, 2 total`を要求しています。部分実行（例：1件だけ通した状態）では合計数が一致しないため、評価者に見抜かれずに完了することはありません。
- **ガードレール**：テストを書き換えて「通す」ことを防ぐため、`tests/` 配下の変更禁止と、`git diff --stat` による差分範囲の可視化を毎ターン要求しています（ナレーションだけでは評価者は違反を検知できないため、目に見える証拠を必須にしています）。
- **停止条件**：ターン数上限とスタック検知を `, または` で主条件に接続する形にしています（独立した文にすると評価者が完了条件として認識しないことが確認されているため）。

補足：`/goal` は自律ループの「次のターンへ進むか」を判定するだけで、各ターン内のツール承認（ファイル編集やコマンド実行）は別問題です。毎ターンの承認プロンプトも省略したい場合は、`/goal` に加えて Auto-Accept Edits モード（Shift+Tab で切り替え）か `--dangerously-skip-permissions` / bypassPermissions 設定を有効にしておく必要があります。`/goal` 自体は承認を自動化しません。
