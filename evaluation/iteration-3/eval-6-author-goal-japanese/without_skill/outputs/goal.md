リポジトリ（`node-app`）を確認しました。`package.json` に `"test:auth": "jest tests/auth"` という専用スクリプトがあり、対象は `tests/auth/login.test.ts`（現在2テスト、`src/auth/login.ts` を検証）でした。これを基に `/goal` の条件を組み立てました。

以下をそのまま貼り付けてください。

```
/goal tests/auth 配下のテストがすべて通る状態にする。証明として、直近のターンで `npm run test:auth`(= `jest tests/auth`) を実行し、その出力に "Tests:" 行が "0 failed" を含み、かつ合計テスト数(現在は2件)が全て passed になっていることを示す — それができない場合、または同じ失敗が2ターン連続で再発した場合は15ターンで停止し、何がブロッカーになっているかを要約する。制約: tests/ 配下のテストファイル(login.test.ts など)は変更・削除しない。各ターンで `git diff --stat` を示し、変更が src/ 配下のみであることを証明すること。
```

補足:

- **検証コマンドの根拠**: `package.json` に `test:auth`(`jest tests/auth`)という専用スクリプトが実在するのでそれを指定しました。全体の `npm test`(`jest`)ではなく認証テストだけに絞ったスクリプトがあったため、そちらを優先しています。
- **エビデンス**: Jest のサマリー行（`Tests: X passed, X total` / `0 failed`）を毎ターン画面に出させることで、評価用モデル(Haiku)がテキストだけで合否を判定できるようにしています。
- **ガードレール**: テストファイルを書き換えて「無理やり通す」ことを防ぐため、`tests/` 配下は変更禁止とし、`git diff --stat` を毎ターン提示させて `src/` 配下のみの変更であることを可視化しています。
- **停止条件**: 15ターンの上限、または同じ失敗が2回連続した場合に停止し、ブロッカーを要約させる形にしています（無限ループ防止）。

**承認プロンプトについて**: `/goal` 自体は自動承認機能ではなく、あくまで「ターンごとに完了条件を評価して続行/終了を判断する」ループです。毎ターンの承認(コマンド実行やファイル編集の許可)を本当に省略したい場合は、`/goal` とは別に Claude Code のセッションを `acceptEdits` または `bypassPermissions` モードで起動する必要があります。通常のデフォルト権限モードのままだと、`/goal` を設定していても `npm run test:auth` の実行やファイル編集のたびに承認を求められます。
