リポジトリを確認しました。認証関連は `tests/auth/login.test.ts`（2テスト）と `src/auth/login.ts` のみで、`package.json` に `test:auth": "jest tests/auth"` という専用スクリプトがあります。これを検証コマンドとして使うのが最も確実です（`npm test` だと無関係なテストまで含みうるため、スコープを絞れる `test:auth` を採用しました）。

以下の `/goal` をそのまま貼り付けてください。

```
/goal tests/auth 配下の全テストが成功している。証明として、直近のターンで
`npm run test:auth` を実行し、その出力の最後のサマリー行に失敗0件(0 failed)
と合計2件(2 total)が表示されていることを示すこと(サブセット実行や過去ターンの
結果の使い回しは不可、必ず直近ターンで再実行すること) — または15ターンに達し
たら停止、もしくは同じ失敗が2回連続で再現したら停止し、その場合は何が障害に
なっているかを要約すること。制約: tests/ 配下のファイルは一切変更・削除しない
こと。各ターンで `git diff --stat` を示し、変更が src/ 配下のみであることを
証明すること。各ターンでターン番号(turn k of 15)を明記すること。
```

補足:
- 検証コマンドは `npm run test:auth`（= `jest tests/auth`）。現状 `tests/auth` にはテストが2件しかないため「2 total」を明示させ、部分実行で条件を満たせないようにしています。
- ガードレールとして、テストファイルの改変で無理やり通す抜け道を塞ぐため `git diff --stat` の提示を毎ターン必須にしています(ナレーションだけでなく実際の差分を評価器が読むため有効です)。
- 「毎ターンの承認はしたくない」とのことなので、`/goal` はターン開始時の停止(Stopフック)は自動化しますが、ターン内のツール承認自体は別問題です。承認ダイアログで止まらないよう、`/goal` を実行する前にセッションを `Auto-accept edits`(shift+tab) または `--permission-mode bypassPermissions`(headless実行なら `claude -p "/goal ..." --permission-mode bypassPermissions` 相当)にしておいてください。これをしないと、`/goal` はターン開始のトリガーにはなっても、各ターン内のコマンド実行確認プロンプトは別途出ます。
