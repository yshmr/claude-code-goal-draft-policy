README.md を確認しました。現状は mTLS / envoy sidecar / JWT claims propagation / OPA bundle など、かなりエンジニア向けの用語が並んでいます。

ただ、そのままだと一つ問題があります。「エンジニアじゃないメンバーが読んでも分かる」は、Claude が自走ループの中で機械的に判定できる条件ではありません。`/goal` はターン終了ごとに「完了条件を満たしたか」を自己判定して継続 or 終了を決める仕組みなので、条件が主観的だと

- 早期に「もう十分分かりやすい」と自己満足して止まる
- 逆にいつまでも「まだ非エンジニアには難しいかも」と言い続けて終わらない（无限ループ）

のどちらかに転びやすいです。実際に確認できる代理条件に落とし込むことをおすすめします。具体的には、書き直したあとの README が満たすべき「観測可能な」基準を並べる形です。例えば：

- 専門用語（mTLS, JWT, OPA, sidecar, SLO など）が出てくる箇所には、その都度平易な一言説明が添えられている
- コマンドや設定値の羅列だけでなく、「これは何のためのサービスか」「困ったときに何をすればいいか」が平文の文章で書かれている
- 見出し構成が「概要 → 使い方 → 困ったときは」のような、非エンジニアでも迷わない順序になっている
- 略語・頭字語が初出時に展開されている（例: SLO = Service Level Objective）

これらは Claude が自分でチェックリストとして読み返せる基準なので、「分かるようになるまで」よりも自走ループに向いています。

以下の完了条件でセットアップすることを提案します。このまま実行してよければ教えてください（必要なら文言は調整します）。

```
/goal README.md を非エンジニアにも読める形に書き直す。完了条件: (1) mTLS, JWT, OPA, sidecar, SLO, RFC 7519 など専門用語には初出箇所で平易な説明が併記されている、(2) 「このサービスは何をするものか」「困ったときにまず何を確認すればよいか」が箇条書きコマンドではなく文章で説明されている、(3) 見出しが 概要 → 使い方 → トラブルシューティング の順に並んでいる、(4) 略語は初出時にフルスペルを併記している。上記4点をすべて満たしたら完了として終了する。
```

補足:
- 「エンジニアじゃないメンバーが読んでも分かるか」を最終的に判断できるのは実際にその人たちだけなので、この /goal は「分かりやすくするための具体的な編集」を最後まで自走させるものであり、本当に理解できたかの確認は別途、実際の非エンジニアメンバーにレビューしてもらうことをおすすめします。
- 条件をこの4点で進めてよいか、あるいは他に外してほしくない基準（例:「このサービスがないと何が困るか」を書く、図を入れる、等）があれば教えてください。
