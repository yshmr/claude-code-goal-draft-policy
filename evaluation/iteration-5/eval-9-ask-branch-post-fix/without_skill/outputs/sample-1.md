リポジトリを確認しました（`docs-repo/README.md` のみ、テストやビルドコマンドは無いシンプルな README1枚のリポジトリです）。現状はこんな内容で、専門用語がかなり多いです。

> mTLS termination via envoy sidecar, JWT claims propagation (RFC 7519), circuit-breaking per upstream SLO budget… helm install w/ values-prod.yaml… OPA bundle rev… GW_JWT_AUD…

「非エンジニアが読んでも分かる」は主観的な基準なので、そのままだと自動判定（`/goal` の評価は会話の文字起こしだけを見る小さなモデルが行うため、実際にコマンドやテストを実行して確認することはできません）が「完了」を正しく判定できません。そこで、判定できる形に落とし込んだうえで `/goal` を設定しました。

## セットアップした /goal

```
/goal README.md が非エンジニアのメンバーにも理解できる内容に書き直されている。以下の基準をすべて満たすことで判定する。判定は必ず「直近のターン」で取得した最新の README.md の内容に対して行うこと。

1. README.md の冒頭付近に、平易な言葉で書かれた概要見出し（例: "## 概要" "## これは何？" "## Overview" のいずれか）が存在する。直近のターンで `grep -nE "^## (概要|これは何|Overview|TL;DR)"  README.md` を実行し、1件以上ヒットすることを示す。

2. README.md に "## 用語集" または "## Glossary" という見出しのセクションがあり、そこに次の専門用語がすべて平易な説明付きで載っている: mTLS, JWT, envoy, sidecar, circuit-breaking, SLO, OPA, helm, configmap, thundering herd, port-forward, 503, RFC 7519。直近のターンで、用語集セクション以降の本文を抽出し、上記の用語リストを1つずつ照合して「見つからない用語」を出力するチェック（例: シェルのfor文で各用語をgrepし、ヒットしなければ MISSING と出力)を実行し、MISSING の件数が 0 件であることを示す。

3. 直近のターンで `cat README.md` を実行し、書き直し後の全文を出力する（内容が実際に変わっていることを目視確認できるようにする）。

制約: 設定値名（GW_UPSTREAM_TIMEOUT_MS 等の環境変数名やデフォルト値、注意書き「Do not set retry budget >0.3 in prod」など）は削除・改変せず、技術的な正確さは保ったまま、用語集や概要で意味を補足すること。各ターン `git diff --stat` を示し、変更が README.md のみであることを示す。

停止条件: 15ターンを超えた場合、または「MISSING な用語の件数」が2ターン連続で減少しない場合は、そこで止めて残っている問題点を要約して報告する。各ターン「turn k of 15」のようにターン番号を明示すること。
```

## この条件にした理由

- **終了条件を数値化**: 「分かるように」という主観的な表現を、(a) 平易な概要見出しの有無、(b) 専門用語がすべて用語集で説明されているか、という2つの grep で確認できる条件に変換しました。
- **検証コマンド**: このリポジトリには package.json やテストスクリプトが無い（README 1枚だけのリポジトリ）ため、`grep` ベースのチェックを証拠として使っています。
- **ガードレール**: 設定値や警告文など技術的に重要な情報を「わかりやすくする」過程で消してしまわないよう明記し、`git diff --stat` で README.md 以外を触っていないことも毎ターン示させます。
- **停止条件**: 用語集が全部埋まらない・見出しが付けられない等で無限ループしないよう、15ターンの上限と「進捗が2ターン止まったら打ち切り」を両方入れています。

貼り付けてそのまま使えます。実行するとClaudeが自動でシェルコマンド（grep等）を叩くので、承認プロンプトを都度出したくない場合は `acceptEdits` または `bypassPermissions` モードで進めることをおすすめします。

なお、用語集に入れる用語リストは今のREADMEの内容から拾ったものです。書き直しの過程で新しい専門用語が増えた場合は、その用語もリストに追加してから再度 `/goal` を設定し直してください（判定は現在リストにある用語だけを見ます)。
