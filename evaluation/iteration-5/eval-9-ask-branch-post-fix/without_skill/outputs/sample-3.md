リポジトリを確認しました。現在の README (`evaluation/iteration-4/fixtures/docs-repo/README.md`) はこんな内容です。

> mTLS termination via envoy sidecar、JWT claims propagation (RFC 7519)、circuit-breaking per upstream SLO budget、helm install、OPA bundle rev...

たしかにこれは非エンジニアには厳しい内容です。ただ、`/goal` をそのまま「非エンジニアが読んで分かるまで自走で書き直し続けて」という条件で設定するのは避けたいと思います。理由は、"分かるかどうか" は最終的に人間の主観判断であり、Claude自身がそれを客観的に検証できる条件になっていないと、以下のどちらかに陥りやすいからです。

- Claudeが「もう十分平易だ」と自己判断して早期に停止してしまう
- 逆に「まだ足りないかもしれない」と際限なく書き直しを続けてしまう(無限ループ、または大量のトークンを消費するだけの空転)

そこで、`/goal` を設定する前に、"完了" を機械的に判定できる基準に落とし込みたいです。以下のような形にするのはいかがでしょうか。

**案: チェックリスト方式**

README が満たすべき条件をあらかじめ具体的にリストアップし(例: 専門用語には日本語での一言説明を併記する / このサービスが何をするものかを一文目で説明する / Quickstart のコマンドの意味を説明する / 略語(mTLS, JWT, SLO, OPA など)を初出時に展開する、など)、Claudeにはそのチェックリストを全項目満たすまで書き直しを繰り返させる、という形です。この方式なら「満たしている/いない」を機械的に確認できるので、自走ループの終了条件として機能します。

もしこの方向でよければ、具体的なチェックリスト項目を一緒に決めてから `/goal` コマンドを組み立てます。逆に「大枠でいいのでとにかく非エンジニア向けに書き直させたい」ということであれば、多少ゆるくなりますが以下のような `/goal` を設定することもできます。

```
/goal README.md を非エンジニアのメンバーが読んで「このサービスが何をするもので、どう使うか」を理解できる状態にする。具体的には: (1) 専門用語(mTLS, JWT, SLO, OPA, envoy sidecarなど)が初出時に平易な説明付きで書かれている、(2) Quickstartの各コマンドが何をするか一文で説明されている、(3) エンジニア以外の第三者(自分自身がその立場になったつもりで)がREADMEだけを読んで概要を説明できると判断できる。この3条件を満たしたら完了とする。
```

ただし正直に言うと、この案でも (3) は結局Claudeの自己判断に依存する部分が残ります。完全に客観的な条件にしたい場合は、上記のチェックリスト方式を先に固めることを強くおすすめします。

どちらの方向で進めますか。チェックリストを一緒に詰めるか、上記のゆるめの `/goal` でまず走らせてみるか、教えてください。
